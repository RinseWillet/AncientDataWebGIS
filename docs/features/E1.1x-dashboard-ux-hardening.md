# E1.1x — Dashboard UX Hardening

**Status:** backlog  
**Priority:** high — required before NAS test-environment deploy  
**Epic parent:** E1 Research Dashboard  
**UX scope:** layout, graph selection, readability, responsiveness, accessibility  
**Out of scope:** visual theme, branding, color system (→ E5 Synthwave)

## Target hardware

| Device | Viewport / context | Priority |
|---|---|---|
| Fairphone 5 | ~393 px CSS width, Chrome Mobile, touch input | **Primary mobile** |
| 15-inch laptop | ~1280–1440 px, desktop browser | **Primary desktop** |
| 32-inch external screen | ~2560 px (likely 1920 px browser window), desktop browser | **Secondary desktop** |

**Implications for layout decisions**
- Fairphone 5 screen is tall and narrow → single-column stacking must be readable without horizontal scroll.
- Touch targets (retry button, nav links) must be ≥ 44×44 px.
- On 32-inch at full-width, the two-column layout must not stretch to an unreadable line width; cap the dashboard `max-width` (e.g., `1400 px centered`).
- Charts must remain legible at both extremes: compact on FP5, not oversized on the 32-inch.
- Test breakpoints to explicitly target: **393 px** (FP5), **768 px** (tablet reference), **1280 px** (laptop), **1920 px** (external screen).

---

## Context

After E1-1/E1-2/E1-3 (endpoint + charts merged), the dashboard is functional but needs
hardening before a deployable test image is published. The main concerns:

1. **Layout** — all charts stack vertically beneath each other; no clear grouping by domain.
2. **Graph selection** — some chart types are redundant or too complex for the data shape.
3. **Introduction text** — there is no framing; the dashboard opens cold on raw numbers.
4. **Responsiveness** — not fully verified across mobile/tablet/desktop breakpoints.
5. **Polish** — number formatting, tooltips, empty/error/loading states need tightening.
6. **Accessibility** — no explicit heading hierarchy, no keyboard focus, no screen-reader labels.

## Chart type decisions (rationale per metric)

| Metric | Current | Decision | Rationale |
|---|---|---|---|
| Sites by type | Pie chart (all slices) | **Pie chart — top 5 + "Other" bucket** | Too many slices creates visual noise. Bucketing clarifies the dominant types. |
| Roads by type (count) | Vertical bar chart | **Horizontal bar chart** | Category labels are text strings; horizontal layout avoids label rotation/clipping. |
| Road length total | KPI card | **Keep as KPI card** | Single number, best shown as card metric. |
| Road length by type | Vertical bar chart | **Horizontal bar chart** | Same reason as roads by type: labels read better horizontally. |
| Cumulative road length trend (line chart) | Currently present | **Remove** | Redundant with the road length bar chart; the line chart implies time-series which the data does not have. Remove entirely. |

## Stories

---

### E1.1x-1 — Dashboard layout: two-column domain sections + intro text

**Size:** M  
**Goal:** restructure the page into a clear intro block + Sites quarter + Roads quarter.

**Tasks**
- [ ] Add an introductory `<section>` at the top with a brief static description of the dataset and a note showing when data was last updated.
- [ ] Restructure layout into two side-by-side domain sections: **Sites** (left) and **Roads** (right) on desktop; stacked on mobile (`< 768 px`).
- [ ] Move KPI cards into their respective domain section (not a single row at the top).
- [ ] Remove the cumulative road length trend (line chart) entirely.
- [ ] Keep footer with generated-at timestamp.

**Acceptance criteria**
- [ ] Desktop (≥ 1024 px): Sites and Roads sections render side by side.
- [ ] Tablet (768–1023 px): sections stack; each section takes full width.
- [ ] Mobile (< 768 px): single-column, intro → sites → roads order.
- [ ] KPI cards belong inside their domain section.
- [ ] Line chart is gone; no other charts removed.
- [ ] All layouts verified in browser at **393 px** (Fairphone 5), **768 px** (tablet), **1280 px** (15-inch laptop), **1920 px** (32-inch external screen).

---

### E1.1x-2 — Graph selection: replace and refine chart types

**Size:** M  
**Goal:** swap charts to best-fit types per metric; remove redundant line chart.

**Tasks**
- [ ] Replace Sites by Type pie chart with a **top-5 + "Other" bucketed pie chart** (aggregation in the component, not backend).
- [ ] Replace Roads by Type vertical bar chart with a **horizontal `BarChart`** using `<XAxis type="number"> / <YAxis type="category">` in Recharts.
- [ ] Replace Road Length by Type vertical bar chart with a **horizontal bar chart** (same pattern).
- [ ] Remove cumulative road length line chart.
- [ ] Update tests to reflect new chart structure and removed chart.

**Acceptance criteria**
- [ ] Sites pie shows max 6 segments (top 5 + Other); remaining types collapsed into Other.
- [ ] Roads count bar chart is horizontal; labels not clipped or rotated.
- [ ] Road length bar chart is horizontal; km values visible in tooltip.
- [ ] No redundant line chart present in DOM or tests.
- [ ] `npm run test:run` passes.

---

### E1.1x-3 — Data readability and number formatting

**Size:** S  
**Goal:** consistent, readable metric display.

**Tasks**
- [ ] Format large numbers with locale-appropriate separator (e.g., `1,234`).
- [ ] Format km totals to 2 decimal places everywhere (KPI card + tooltips).
- [ ] Normalize category display to title case in chart labels/tooltips (e.g., `roman road` → `Roman Road`).
- [ ] Tooltip content should show both the label and the value with unit (e.g., `Roman Road: 800.00 km`).

**Acceptance criteria**
- [ ] `1234.56 km` → `1,234.56 km` in KPI card.
- [ ] All chart tooltips show human-readable label + formatted value with unit.
- [ ] Type labels are title-cased visually in the UI (backend normalization to lowercase remains unchanged).
- [ ] No raw `unknown` visible in charts without a user-friendly label.

---

### E1.1x-4 — Loading/error/empty state polish

**Size:** S  
**Goal:** every state is intentional and actionable.

**Tasks**
- [ ] Replace spinner-only loading with a structured skeleton (KPI card placeholders + chart area placeholder boxes that match the real layout).
- [ ] Error retry button calls `fetchDashboard()` directly; does not reload the full page (`window.location.reload()`).
- [ ] Empty state provides a brief explanation (e.g., "No data has been loaded into the database yet").
- [ ] State transitions do not cause visible layout shifts (no jump when data loads).

**Acceptance criteria**
- [ ] Loading state shows card + chart area placeholders matching final layout shape.
- [ ] Clicking Retry re-fetches without full page reload.
- [ ] Empty state message is present and clear.
- [ ] No layout shift between loading → loaded transition.

### E1.1x-5 — Accessibility baseline

**Size:** M  
**Goal:** dashboard is usable without relying only on color or visual context.

**Tasks**
- [ ] Add `<main>`, `<section>`, `<h2>` landmark structure.
- [ ] Each chart section has a visible heading.
- [ ] KPI card values have `aria-label` with unit (e.g., `aria-label="Total sites: 42"`).
- [ ] Retry button has accessible name (`aria-label` or visible text).
- [ ] Interactive controls are keyboard-focusable with a visible focus ring.
- [ ] Ensure no information is conveyed by color alone (labels/values always present).

**Acceptance criteria**
- [ ] Heading hierarchy: `h1` → page title, `h2` → Sites / Roads section headings.
- [ ] KPI cards pass basic aria review.
- [ ] Retry button accessible by keyboard (Tab + Enter).
- [ ] `npm run lint` passes with no new a11y-related warnings.
