# E2.0 — Media Storage Foundation and Operator Runbook

**Status:** draft  
**Epic parent:** E2 Photo & Media Integration  
**Priority:** high (prerequisite for E2-1+)

---

## Why this story exists

AncientData is moving from text/geometry records to media-bearing records (photos of sites and roads). That introduces a new operational concern: where binaries live, how they are served, and how they are backed up.

This story answers that question concretely for **your current stack** and documents the decisions as a runbook future implementers can follow.

---

## Your current stack (as observed in `docker-compose.yml`)

Your Docker Compose stack runs on a **Synology NAS**. This is visible from the bind-mount paths:

```
/volume1/docker/postgis_pgadmin_stack/postgis    → PostGIS data
/volume1/docker/postgis_pgadmin_stack/geoserver  → GeoServer data dir
/volume1/docker/postgis_pgadmin_stack/postgisadmin → pgAdmin data
```

**Key insight:** you are already using a simple, reliable pattern — each service gets its own named subdirectory under `/volume1/docker/postgis_pgadmin_stack/`. Media files fit naturally into the same pattern.

---

## What "external storage" means in your case

**Short answer: a directory on your NAS, exposed to the Spring Boot container via a Docker bind mount.**

You do not need to set up a separate server, S3 bucket, or object-store daemon for this project phase. The same mechanism already used for PostGIS and GeoServer works for photos.

Here is what happens end to end:

```
Browser / frontend
      │
      │  GET /api/media/files/{filename}
      ▼
Spring Boot container   (port 8080)
      │
      │  reads file from /app/media/  (inside container)
      │
      ▼
  bind mount
      │
      │  maps to  /volume1/docker/postgis_pgadmin_stack/ancientdata_media/
      ▼
NAS filesystem          (persistent, backed up by Synology HyperBackup)
```

You upload a photo through the API → it lands on the NAS directory → the Spring Boot container serves it back through a static file endpoint → the frontend `<img>` tag loads it by URL.

---

## Why not store images in Postgres?

- Your PostGIS container is also used by QGIS and pgAdmin. Storing large binaries in the DB puts pressure on everyone sharing that database.
- Postgres `bytea` blobs are not optimised for serving image content — you cannot stream them through a CDN or cache them at the HTTP level easily.
- Backup is simpler when images are files: you can back up the NAS directory independently without a DB dump.

Why not set up MinIO/S3 now?

- Adding another container adds operational complexity (credentials, ports, bucket lifecycle) that is not needed for a personal NAS deployment.
- The bind-mount filesystem approach is easier to understand, debug, and restore.
- The backend will be written with a `MediaStorageService` interface, so swapping to MinIO/S3 later is a one-file change without touching the rest of the application.

## Recommended storage layout on NAS

Create a new subdirectory alongside existing stack data:

```
/volume1/docker/postgis_pgadmin_stack/ancientdata_media/
    site/
        {site_id}/
            {uuid}.jpg
            {uuid}.webp
    road/
        {road_id}/
            {uuid}.jpg
```

This mirrors your existing pattern and keeps all stack data in one Synology share.

## Changes needed in `docker-compose.yml`

Add a bind mount to the `ancientdata` service:

```yaml
  ancientdata:
    image: rinsedev/ancientdata
    restart: on-failure
    depends_on:
      - db
    ports:
      - 8080:8080
    volumes:
      - /volume1/docker/postgis_pgadmin_stack/ancientdata_media:/app/media:rw
    environment:
      - DATABASE_HOST=db
      # ... existing env vars
      - MEDIA_STORAGE_PATH=/app/media
      - MEDIA_BASE_URL=http://<your-nas-ip>:8080/api/media/files
```

The container writes to `/app/media` (inside the container), which is the NAS directory on the host.

## What to store in the DB (`media_asset` table)

The NAS holds the binary file. The database holds everything else:

| Column | Type | Notes |
|---|---|---|
| `id` | bigserial | PK |
| `target_type` | varchar(10) | `ROAD` or `SITE` |
| `target_id` | bigint | matches road/site id |
| `storage_key` | varchar(512) | relative path from media root, e.g. `site/42/abc123.jpg` |
| `mime_type` | varchar(64) | `image/jpeg`, `image/png`, `image/webp` |
| `file_size_bytes` | bigint | |
| `caption` | varchar(1000) | nullable |
| `author` | varchar(255) | nullable |
| `source` | varchar(512) | nullable |
| `license` | varchar(255) | nullable |
| `date_taken` | date | nullable |
| `is_cover` | boolean | default false |
| `visibility_status` | varchar(20) | `PENDING`, `APPROVED`, `REJECTED`, `HIDDEN` |
| `created_at` | timestamptz | |
| `updated_at` | timestamptz | |
| `created_by` | varchar(255) | nullable, username |

The `storage_key` is the only link between the DB record and the file. If you know the key, you can build the full URL as `{MEDIA_BASE_URL}/{storage_key}`.

## URL strategy

Use **simple public stable URLs** for this phase:

```
http://<nas-ip>:8080/api/media/files/site/42/abc123.jpg
```

The Spring Boot backend will serve files from its media directory through a dedicated controller endpoint. This means:

- No signed URL complexity.
- Easy to cache with a reverse proxy later.
- Works the same whether accessed from inside or outside the LAN.

---

## API shape

```
POST   /api/media           multipart/form-data (file + metadata fields)   [auth required]
GET    /api/media?targetType=SITE&targetId=42                               [public]
GET    /api/media/files/**  static file serving                              [public]
PATCH  /api/media/{id}      update metadata / visibility                    [admin]
DELETE /api/media/{id}      delete record + file                            [admin]
```

## Upload guardrails

- Accepted MIME types: `image/jpeg`, `image/png`, `image/webp`
- Max file size: **10 MB** per file (already supported by `spring.servlet.multipart.max-file-size=50MB` in `application.properties` — can tighten)
- Filename: **always regenerate as UUID** server-side; never trust the client's filename
- Extension: derive from detected MIME type, not from the original filename

## Backup and restore strategy

### Primary backup: Synology HyperBackup (or local NAS snapshots)

Your NAS's built-in backup (HyperBackup or equivalent) already covers `/volume1/docker/` automatically. This is your main recovery path.

For a complete restore:

1. Restore the NAS directory `/volume1/docker/postgis_pgadmin_stack/ancientdata_media/` from NAS backup.
2. Restore the PostGIS DB (which contains the `media_asset` metadata).
3. Verify: query `media_asset` and confirm at least one `storage_key` file is present in the restored directory.

### Offsite backup: Google Drive sync (optional, recommended)

You can push a read-only copy of the media files to Google Drive as an extra layer of protection.

**Why:** if the NAS fails catastrophically and backups are somehow unavailable, you have an offsite copy. Google Drive is free and reliable.

**How:** periodically sync the media directory to Google Drive using `rclone`. This is a one-time setup:

```bash
# 1. Install rclone (on your NAS or a Linux machine that can reach it)
#    Instructions: https://rclone.org/install/

# 2. Configure Google Drive remote in rclone (interactive, one time)
rclone config

#    When asked, choose:
#    - name: "gdrive"
#    - type: 16 (Google Drive)
#    - follow OAuth flow to authorize rclone

# 3. Add a cron job (on NAS or Linux box) to sync weekly
crontab -e

#    Add this line (syncs every Sunday at 2 AM):
0 2 * * 0 rclone sync /volume1/docker/postgis_pgadmin_stack/ancientdata_media/ gdrive:/AncientDataMediaBackup --exclude-if-modified-before 24h -v >> /var/log/rclone_sync.log 2>&1
```

The `--exclude-if-modified-before 24h` flag means only files changed in the last day are synced, which keeps bandwidth low after the first full backup.

**To restore from Google Drive:**

```bash
rclone copy gdrive:/AncientDataMediaBackup /volume1/docker/postgis_pgadmin_stack/ancientdata_media/
```

Then restart the Spring Boot container and verify the app can load the images.

This setup costs nothing (under Google Drive's free tier for most projects) and takes ~10 minutes to set up once.

## Schema change path

Because Flyway is disabled and schema is externally managed:

1. Write the DDL in `docs/architecture/sql/V_media_asset__create_table.sql`.
2. Review and apply via pgAdmin or `psql` directly against the PostGIS container.
3. Update the Spring Boot entity, repository, service, and controller after the table exists.

---

## Moderation and visibility policy

- Default visibility on upload: `PENDING`
- Public GET endpoints return only `APPROVED` records
- Admin PATCH can move to `APPROVED`, `REJECTED`, or `HIDDEN`
- No public upload path (admin-only upload to start; can be relaxed later)
